### Locals
- Variables bring data in, Locals organize it, Outputs send it out.
- Add a locals block when you want to reuse an expression throughout your configuration. Like a Name, Tag, any value | Helps avoid repition and give a meaningful name.
- Use the local.<NAME> syntax to reference values from a locals block in your configuration
- You can access local values in the module where you define them, but not in other modules. However, you can pass a local value to a child module as an argument.
```tf
# Use local values in situations where you either reuse a single value in many places to let you change a value in a single place or when the value is the result of a complex expression.
locals {
  prefix = "${var.project}-${var.environment}"
}

resource "aws_s3_bucket" "logs" {
  bucket = "${local.prefix}-logs"
}

resource "aws_s3_bucket" "backup" {
  bucket = "${local.prefix}-backup"
}

resource "aws_iam_role" "app" {
  name = "${local.prefix}-role"
}
```
### Meta arguments
> that control how Terraform creates and manages your infrastructure. You can use meta-arguments in any type of resource. You can also use most meta-arguments in module blocks.
But what if you want to:
```tf
resource "aws_instance" "web" {
  ami           = "ami-123"
  instance_type = "t2.micro"
}
```
- Create 5 instances instead of 1?
- Create resources conditionally?
- Specify that one resource depends on another?
- Prevent accidental deletion?
- Use a different provider configuration?

> These behaviors aren't resource properties—they're Terraform behaviors. That's what meta-arguments control
