| Aspect                        | Mutable Infrastructure                                                                         | Immutable Infrastructure   (Terraform)                                                                                              |
| ----------------------------- | ---------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| **Definition**                | Existing servers are updated and modified in place.                                            | Existing servers are never changed; new servers are created to replace them.                                             |
| **Deployment**                | Apply updates directly to the running server.                                                  | Build a new server/image with updates, deploy it, then retire the old one.                                               |
| **Versioning**                | Can result in many intermediate states (e.g., partially updated servers).                      | Maintains clear, discrete versions (e.g., V1 or V2, never "V1.5").                                                       |
| **Risk**                      | Higher risk of configuration drift and inconsistent server states if updates fail.             | Lower risk because each deployment is a fresh, validated instance.                                                       |
| **Troubleshooting**           | More difficult due to differences between servers over time.                                   | Easier because every server of the same version is identical.                                                            |
| **Rollback**                  | May require reversing in-place changes, which can be complex.                                  | Simple—redirect traffic back to the previous version and remove the new one.                                             |
| **Data Handling**             | Local data remains on the server during updates.                                               | Application data should be externalized (e.g., shared database or persistent storage) so servers can be safely replaced. |
| **Infrastructure Complexity** | Complexity increases as servers gradually diverge from one another.                            | Simpler to manage because infrastructure consists of known, consistent versions.                                         |
| **Best For**                  | Stateful systems or infrequently updated components (such as some databases).                  | Stateless applications, cloud-native workloads, and frequent deployments.                                                |
| **Main Trade-off**            | Faster, simpler updates without replacing infrastructure, but greater risk of inconsistencies. | More reliable and predictable deployments, but requires replacing infrastructure and externalizing persistent data.      |

## Modules
> When you repeatedly provision collections of resources with similar configuration
- helps you standardize how you provision infrastructure and lets you quickly and predictably provision the resources you need.
- To define a module, create a new directory for it and place one or more .tf files inside
```sh 
Root Module  # My main code folder
├── module.web1
│   └── module.sg
└── module.web2
    └── module.sg
```
- Terraform can load modules from multiple sources, including the local file system, a Terraform registry, and VCS repositories.
```tf
module "vpc" {
  source = "git::https://example.com/vpc.git?ref=v1.2.0"
}
```
- **HCP Terraform and Terraform Enterprise** include a  `private module registry` for sharing modules internally within your organization
- Else use Terraform registry to get modules.
- Terraform Module = Input Variables + Resources + Output Values
-  You can reference the values in the parent module using the `module.<MODULE-NAME>.<OUTPUT-NAME>` expression.
-  You can also remove a module from your Terraform state without destroying the real infrastructure objects it manages:

1. Replace the module block from your configuration with a removed block.
2. And add the lifecycle block to your removed block.
3. Add the destroy argument and set it to `false.
The following example removes the module.example resources from the Terraform state without destroying them:
```tf
removed {
  from = module.example

  lifecycle {
    destroy = false
  }
}
```
## Variables
- You can use variables or locals in Module Source but add word `const = true`
```tf
module "consul" {
  source = var.module_source
}

variable "module_source" {
  type  = string
  const = true
}
```
### Terraform Best Practices
- `moved blocks` let you safely `reorganize Terraform code` by telling Terraform where an existing resource moved, preventing unnecessary destroy-and-create operations.
```tf
resource "aws_instance" "server" {}

moved {
  from = aws_instance.web # old resource block
  to   = aws_instance.server #new resource
}
```
- Configuration validation = check if something important is true before using it. If not, show a clear error.
- Instead of one giant module, break everything into building blocks.
```tf
module "network" {
  source = "./modules/network"
}

module "compute" {
  source = "./modules/compute"          

  vpc_id = module.network.vpc_id           # Now Module B cannot work without Network Module They are tightly coupled.
                                         # AKA Dependency Inversion
}

module "database" {
  source = "./modules/database"

  subnet_ids = module.network.private_subnet_ids
}
```
- The root module is responsible for composing those blocks by passing outputs from one module into another, rather than having modules create or discover all of their own dependencies. This approach solves problems of tight coupling, poor reusability, duplicated infrastructure,

## Workspace = RG in Azure = folder in TF
> The persistent data stored (like state) in the backend belongs to a workspace. The backend initially has only one workspace containing one Terraform state associated with that configuration.
- Working locally or in your own CI/CD pipeline (e.g., GitHub Actions, GitLab CI, Jenkins), the `free Terraform CLI workspaces` are sufficient.
- Managing infrastructure across a team and want centralized state, collaboration, and managed runs, `HCP Terraform workspaces` may be a better fit.
A common misconception is that terraform workspace requires Terraform Cloud. It does not—the CLI workspace feature has always been part of the free Terraform CLI.
- Terraform starts with a single, default workspace named default that you cannot delete. If you have not created a new workspace, you are using the default workspace in your Terraform working directory.

## Drift Detection Terraform:
- If drift is detected, you should revert the manual changes to match the Terraform configuration, update the configuration to reflect the new desired state, and run terraform apply to bring the infrastructure back into alignment with the configuration.

## Terraform Provisioners
> Provisioners execute scripts or commands on a resource after it is created. They are often used for tasks like configuring VMs or installing software. 

There are two types of provisioners:
- Local provisioners, which run on the machine executing Terraform.
- Remote provisioners, which run on the target resource via SSH or WinRM.

Since provisioners introduce dependencies and reduce Terraform's declarative nature, they should be used sparingly. An alternative is to use configuration management tools like Ansible or cloud-init.

## Dependency in terraform
- Explicit dependencies: `depends on block`: Used when terraform auto dependency won't work
- Implicit dependencies:  are created when one resource references another in its configuration (for example, referencing an aws_security_group in an aws_instance)

## Terraform Import
Starting form terraform 1.6 : `terraform plan -generate-config-out=generated.tf`
So instead of writing hundreds of lines yourself, Terraform generates a starting point. You should still review and clean up the generated configuration before using it.
Great example. Let's assume:

* **Subscription:** `abc`
* **Resource Group:** `123`
* Terraform version: **1.6 or later**
* Provider: Azure (`azurerm`)

I'll show you the complete process from scratch.

---

**step 1: Create a project folder**

```
terraform-import/
│
├── providers.tf
├── main.tf
├── import.tf
└── terraform.tfstate   (created later)
```

Initially you only create these three files.

---

**Step 2: Write the provider configuration**

**providers.tf**

```hcl
terraform {
  required_version = ">= 1.6"

  required_providers {
    azurerm = {
      source  = "hashicorp/azurerm"
      version = "~> 4.0"
    }
  }
}

provider "azurerm" {
  subscription_id = "abc"

  features {}
}
```

This tells Terraform:

* Use Azure provider.
* Connect to subscription `abc`.

---

**Step 3: Create an empty resource**

Terraform needs to know **where** the imported resource will live in the state.

**main.tf**

```hcl
resource "azurerm_resource_group" "rg" {
}
```

Notice there are **no properties yet**.

The important part is the resource address:

```
azurerm_resource_group.rg
```

---

**Step 4: Add an import block**

Create **import.tf**

```hcl
import {
  to = azurerm_resource_group.rg

  id = "/subscriptions/abc/resourceGroups/123"
}
```

The ID is the Azure Resource Manager ID.

---

**Step 5: Initialize Terraform**

Run

```bash
terraform init
```

Terraform downloads the Azure provider.

---

**Step 6: Generate configuration**

Run

```bash
terraform plan -generate-config-out=generated.tf
```

Terraform does two things:

* Imports the resource
* Creates a new file called

```
generated.tf
```

It may look like this:

```hcl
resource "azurerm_resource_group" "rg" {
  location = "East US"
  name     = "123"

  tags = {
    Environment = "Production"
  }
}
```

You didn't write this.

Terraform generated it.

---

**Step 7: Review generated.tf**

Check every value.

Maybe Azure has

```
Location = East US

Tags

Owner = IT

Department = Cloud
```

Terraform should show the same values.

If something looks wrong, edit it.

---

**Step 8: Move the generated code**

Once you're happy, copy the resource into **main.tf**.

Example:

```hcl
resource "azurerm_resource_group" "rg" {
  name     = "123"
  location = "East US"

  tags = {
    Owner      = "IT"
    Department = "Cloud"
  }
}
```

You can now delete

```
generated.tf
```

---

**Step 9: Run plan again**

```
terraform plan
```

Expected output:

```
No changes.

Your infrastructure matches the configuration.
```

This means Terraform's configuration matches the actual Azure resource.

---

**What happened behind the scenes?**

Initially:

```
Azure

Subscription abc

└── Resource Group
      123
```

Terraform knew nothing.

State:

```
(empty)
```

After import:

State:

```
azurerm_resource_group.rg

↓

/subscriptions/abc/resourceGroups/123
```

Terraform now knows:

```
"rg" in my code
        =
Resource Group 123 in Azure
```

---

**Files at the end**

```
terraform-import/

providers.tf

main.tf

terraform.tfstate

.terraform/

.terraform.lock.hcl
```

The temporary files

```
generated.tf
import.tf
```

can be removed after the import is complete if you no longer need them.

---

**What if you use the older `terraform import` command?**

Before import blocks (or if you prefer the CLI), you would:

1. Write the provider configuration.
2. Write the resource block (even if it's empty).
3. Run:

```bash
terraform import azurerm_resource_group.rg /subscriptions/abc/resourceGroups/123
```

This updates only the **state**. It **does not** generate the resource configuration, so you must manually add all the attributes (such as `name`, `location`, and `tags`) to `main.tf`. That's the main advantage of the newer `import {}` block together with `terraform plan -generate-config-out=generated.tf`: it saves you much of that manual work.
